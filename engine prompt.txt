Let's design a business rules engine in Rust. The engine will receive rules data in a JSON format like:

```json
{
    "conditions": {
        "any": [
            {
                "name": "example_rule_1",
                "value": [1, 2, 3],
                "operator": "shares_at_least_one_element_with"
            },
            {
                "name": "example_rule_2",
                "value": [11, 12, 13],
                "operator": "in_list"
            }
        ],
        "all": [
            {
                "name": "example_rule_3",
                "value": 100,
                "operator": "equal_to"
            },
            {
                "name": "example_rule_4",
                "value": [22, 34],
                "operator": "shares_at_least_one_element_with"
            }
        ]
    },
    "actions": [
        {
            "name": "per_diem",
            "params": {
                "rate": 100,
            }
        }
    ],
}
```

The engine should evaluate if the conditions are met when compared to an object of dat and then execute the actions to mutate that object with the parameters provided in the actions as well as the metadata from the conditions.


Assume that the rules are already serialized into a Rust struct. The engine should be able to execute rules across any data structure by implementing the rules as traits on the object we want to compare against.

The rule name (i.e. `example_rule_1`, `example_rule_3`, etc) describes the name of the rule we want to apply, for example it might be `length_of_stay`` or `procedure_code``, which means the rule name indirectly describes the field of the data structure we are comparing against as well as whatever logic might be required to get that data for the given rule.

Other notes:

The engine must be able to evaluate these rules against a set of data.
The rules can be combined using logical operators like "any" and "all".
The engine should also support different types of operators, such as "equal_to", "in_list", and "shares_at_least_one_element_with", with new operators being easy to add.
The engine must be able to handle nested conditions and actions.
The rules engine should be extensible, allowing for new operators and actions to be added easily.
The engine should use as few dependencies as possible and should be efficient in terms of performance.
